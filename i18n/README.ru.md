> **Архитектурная память для рефакторинга.** Обзор радиуса изменений
> по коду, документации и git — на одной интерактивной
> карте, которую читаете и вы, и ваш ИИ-агент. Набор легких утилит и простой, но очень функциональный UI в виде автономного HTML-документа, которым можно отправить "как есть". Легкость и приватность.

`build-graph` рисует **интерактивный граф в одном HTML-файле**, связывающий
пять слоёв, которые не объединяет больше ни один инструмент:

- **код → код** — импорты Python (на основе AST, с учётом `TYPE_CHECKING`)
- **код ↔ документация** — какие markdown-файлы упоминают какие исходники
- **git-дрейф** — оверлей added / modified / renamed / deleted плюс призрачные
  узлы для файлов, которых больше нет
- **тепловая карта изменений файлов** — позволяет отследить и откорректировать наиболее горячие точки изменений, плюс потенциальные источники багов.
- **карта покрытия тестами** — читает проектный coverage.xml и наглядно показывает покрытие кода тестами, позволяет подсветить наименее покрытые файлы.

…и экспортирует ту же карту в **компактный, экономный по токенам JSON**,
рассчитанный на подачу в контекст LLM-агента.

И всё это — с **нулём зависимостей**: чистая стандартная библиотека Python,
`pip install` не тянет ничего дополнительного. Единственный сторонний код — это D3.js в
браузере, подключаемый с CDN с SRI-пиннингом либо полностью встраиваемый
через `--no-cdn` для полной автономности.

**[▶ Live демо](https://mr-freewan.github.io/build-graph/)** — граф этого репозитория (dogfood) с синтетическим оверлеем, чтобы
режимы Git и покрытия тестами тоже были кликабельными.

## Установка

```bash
pip install graph-build        # или: uv tool install graph-build
```

Можно ставить прямо из GitHub:

```bash
pip install git+https://github.com/Mr-Freewan/build-graph.git

# или из клона:
git clone https://github.com/Mr-Freewan/build-graph.git
pip install ./build-graph
```

> Дистрибутив на PyPI называется `graph-build` (прямое имя занято); имена
> устанавливаемых команд остаются прежними: `build-graph`, `find-related-docs`,
> `verify-doc-links`.

## Быстрый старт

```bash
cd your-project
build-graph                    # автообнаружение, конфиг не нужен → docs/graph.html
build-graph --compact          # + docs/graph-compact.json для ИИ-агентов
build-graph --init             # опционально: закрепить найденную структуру в graph.toml
```

Два инструмента-компаньона — `find-related-docs` (обратный поиск: код → доки)
и `verify-doc-links` (проверка битых ссылок для CI) — поставляются в том же
пакете; см. [CLI-утилиты](#cli-утилиты).

## Почему не другие инструменты?

- **pydeps / Import Linter** — только импорты; нет слоя документации, нет
  git-дрейфа.
- **lychee и подобные** — проверяют мёртвые URL; нет карты, нет слоя кода.
- **Граф-вью в Obsidian** — только заметки; не видит ваш код.
- **Repomix / Gitingest** — упаковывают *текст* репозитория для LLM; build-graph
  даёт *структуру*: ~2 % от токенов, которых стоил бы сырой текст (см.
  [цифры](#во-что-это-обходится-по-контексту)).
- **graphify, understand и другие** — тянут за собой много зависимостей, используют недетерминированный LLM для анализа.


## Создан для ИИ-агентов

`--compact` пишет самодокументированный JSON-снимок (встроенная легенда,
индексированные узлы, трёхбуквенные коды типов), который агенты используют для:

1. **Радиуса поражения** — входящие импорты файла, который вы собираетесь
   менять, без grep.
2. **Маршрутизации документации** — какой ADR / справочный документ прочитать
   *перед* правкой файла.
3. **Трёхслойной синхронизации документации** — граф показывает (1) что
   задокументировано, (2) что должно быть задокументировано, но не сделано, и
   (3) что задокументировано, но больше не существует (призрачные узлы =
   детектор устаревания).

Добавьте `build-graph --compact` в pre-commit-хук или шаг CI, чтобы карта была
свежей для каждой сессии агента.

### Компактный формат

`--compact` пишет `graph-compact.json` (схема v2): узлы — индексированный
массив, рёбра — строки `[индекс_источника, индекс_цели, тип, [номера_строк]]`,
трёхбуквенные коды для каждой категории и типа ребра. Ключ `legend` встраивает
полную таблицу расшифровки — агенту не нужна внешняя схема, файл объясняет сам
себя:

```jsonc
{
  "v": "2.0",
  "legend": { "...": "что означает каждое поле и код ниже" },
  "stats": { "nodes": 1070, "ghosts": 0, "edges": 6279 },
  "n": [
    { "p": "smm_bot_async/core/security/access.py", "t": "cor", "d": 56 },
    { "p": "docs/explanation/adr/0009-parser-framework.md", "t": "adr",
      "d": 11, "s": "mod" }
  ],
  "e": [
    [ 1, 75, "d2d", [186] ]
  ]
}
```

`p` — путь, `t` — категория, `d` — степень, `s` — git-статус (опускается,
когда чисто). Типы рёбер: `c2c` импорты, `c2d` упоминания в доках, `d2d`
ссылки между доками, `dcs` ссылки из docstring, `typ` только `TYPE_CHECKING`,
`ren` git-переименования. Удалённые, но всё ещё упоминаемые файлы едут следом
как призрачные узлы (`"G": 1`).

### Во что это обходится по контексту

Реальные цифры с продакшн-репозитория — 1 070 сопоставленных файлов, 6 279
рёбер (токены ≈ байты / 4, обычная грубая оценка):

| Что вы кладёте в контекст            |  Размер | ≈ Токенов  |
|--------------------------------------|--------:|-----------:|
| Сами сопоставленные файлы            |   15 МБ | ~3 700 000 |
| `--json` (подробный снимок)          |  1,6 МБ |   ~410 000 |
| **`--compact`**                      | **0,33 МБ** | **~80 000** |

Вся архитектура — каждый импорт, каждое упоминание в документах, каждая устаревшая
ссылка — укладывается в ~2 % от того, во что обошёлся бы сырой текст, и влезает
в одну сессию с контекстом 200 k с запасом для работы. Без карты агент
переоткрывает эту структуру каждую сессию: десятки спекулятивных grep'ов и
чтений файлов, сжигающих сопоставимое число токенов *на каждый вопрос*, а не
единожды. На небольших проектах карта почти бесплатна — компактный снимок этого
самого репозитория весит 4 КБ ≈ ~1 000 токенов.

<summary>Не верьте этим цифрам на слово — проверьте на своем проекте</summary>

```bash
  raw corpus (1070 files)         14.3 MB    3,757,913     100.0%

  --compact export (schema v2)   311.4 KB       79,729      2.1%
```

### Промпт для старта

```text
graph-compact.json is a dependency map of this repository: nodes are
files, edges are imports and documentation mentions. Read the embedded
"legend" key first — it explains every field and code.

Using the map (before any grep):
1. Lay of the land: the 10 highest-degree hubs, grouped by category,
   with one line each on why they're central.
2. I'm about to modify <path/to/file.py>. List the blast radius:
   direct and 2-hop incoming importers, plus every doc that mentions
   the file — and tell me which of those docs to read first.
3. Anything suspicious: ghost nodes (docs pointing at deleted files),
   zero-degree modules, docs nothing links to.

Verify any surprising claim against the actual source before acting.
```

**Больше промптов:** [Промпты для ИИ-агентов](agent-prompts.ru.md) — готовые
промпты для радиуса поражения, трёхслойной синхронизации документации,
детекции призраков и охоты на мёртвый код.

## Интерактивный граф

- **Canvas-renderer** — плавно на 1000+ узлов / 6000+ рёбер (предпрогретая
  раскладка, отсечение по вьюпорту, LOD подписей).
- **6 типов рёбер** — doc→doc, code→doc, code→code, только-типы
  (`TYPE_CHECKING`), упоминания в docstring, git-переименования.
- **Git-оверлей** — цвета статусов + призрачные узлы + рёбра переименований;
  `--mock-git` для синтетического демо.
- **Diff графа** — `--diff-base REF` сравнивает рабочее дерево с git-ref'ом:
  статусы файлов питают Git-оверлей, новые рёбра зависимостей показаны зелёным,
  а удалённые — красным (пунктиром), со счётчиками в легенде. Добавьте
  `--diff-head REF`, чтобы сравнить два конкретных ref'а вместо рабочего дерева.
- **Heat-оверлей** — цвет узла по частоте git-коммитов (градиент синий→красный),
  по всей истории по умолчанию либо за последние N дней через `--heat-days N`.
  Слайдер min-edits в легенде прячет всё, что холоднее выбранного порога —
  рёбра следуют за узлами. Взаимоисключается с Git-оверлеем — оба перекрашивают
  узлы, поэтому одновременно включён только один; но, в отличие от режима Git,
  он аддитивный: слой Node types остаётся видимым и рабочим под ним.
- **Coverage-оверлей** — цвет узла по покрытию строк тестами (градиент
  зелёный→красный — этот про поиск плохо покрытых файлов, поэтому читается
  наоборот относительно Heat) из Cobertura-файла `coverage.xml`
  (`--coverage PATH`, напр. `pytest --cov=your_pkg --cov-report=xml`). Слайдер
  max-coverage прячет всё, что покрыто *больше* выбранного потолка, изолируя
  худшие по покрытию файлы по мере снижения порога; включение также
  автоматически прячет в легенде все Node types, кроме кода (возвращаются одним
  кликом). Тоже взаимоисключается с Git и Heat. Выключен — и его кнопка скрыта —
  когда отчёт не задан.
- **Инфо узла** — наведите на любой узел, чтобы увидеть его имя и путь; в
  режиме Heat или Coverage — ещё и число правок или процент покрытия за цветом.
  Тултипы рёбер выключаются, пока активен любой из этих режимов.
- **Средства анализа** — кандидаты в мёртвый код, детектор циклов импортов
  (Tarjan SCC по рантайм-импортам; рёбра `TYPE_CHECKING` не считаются), кольцо
  сирот, кратчайший путь между двумя узлами (Shift+клик), изоляция типа,
  исключение по имени. Голое упоминание в доке одноимённого файла (`config.py`
  с десятками совпадений, без пути) кредитуется одному узлу категории
  `ambiguous`, а не разлетается на всех кандидатов.
- **Шаринг** — виды, закодированные в URL (Copy link), экспорт в Mermaid
  сфокусированного подграфа, экспорт полного/компактного JSON.
- **Комфорт** — 10 языков UI, тёмная/светлая темы, согласованные по тону
  пастельная/насыщенная палитры, перетаскиваемые стеклянные панели, глубокие
  ссылки в IDE (VS Code / Cursor / PyCharm), встроенный FAQ (`?`).

Всё укладывается в **один самодостаточный HTML-файл** — приложите его к PR,
отправьте в чате, откройте офлайн.

## Конфигурация (опционально)

Автообнаружение классифицирует каждый отслеживаемый файл по виду (код /
документация / конфиг / локаль) × расположению, определяет ваш пакет и раскладку
документации и генерирует детерминированные цвета. `graph.toml` — это лишь
override:

```bash
build-graph --init           # сгенерировать graph.toml, закрепив текущую структуру
build-graph --init --diff    # доложить о дрейфе (новые папки, устаревшие пины), ничего не менять
build-graph --init --merge   # дописать покрытие новых папок, сохранив ваши правки
```

Аннотированный формат см. в [`graph.example.toml`](../graph.example.toml)
(категории `[docs]`, каталоги `[[code]]`, `[[rules]]`, исключения `[scan]`,
исключения `[dead_code]`, пины цветов).

Два опциональных текстовых компаньона, оба в корне проекта:

- `known-brokens.txt` — белый список ложных срабатываний `verify-doc-links`
  (по одному точному пути на строку).
- `exclude-dirs.txt` — список имён каталогов для пропуска, используется только
  когда git недоступен (с git источник истины — `.gitignore`).

## Справочник CLI (build-graph)

| Флаг | Действие |
|---|---|
| `--root PATH` | корень проекта для сканирования (по умолчанию: cwd) |
| `--config PATH` | расположение graph.toml (по умолчанию: `<root>/graph.toml`) |
| `--output PATH` | вывод HTML (по умолчанию: `docs/graph.html` или `[output].path`) |
| `--scope full\|package` | весь репозиторий (по умолчанию) или только пакет+тесты+доки |
| `--json` / `--compact` | подробный / ориентированный на агента JSON-снимок |
| `--docs-only` / `--no-tests` | урезать набор узлов |
| `--no-cdn` | полностью офлайн-вывод: встроить D3.js инлайн (с проверкой SHA-256) и убрать внешнюю ссылку на шрифт |
| `--mock-git` | синтетический git-оверлей для демо/тестов |
| `--diff-base REF` | ref-diff-сборка: статусы + изменения рёбер относительно git-ref'а (head = рабочее дерево, если не задан `--diff-head`) |
| `--diff-head REF` | с `--diff-base`: сравнивать с этим ref'ом вместо рабочего дерева |
| `--heat-days N` | ограничить Heat-оверлей последними N днями (по умолчанию: вся история) |
| `--coverage PATH` | включить Coverage-оверлей из Cobertura-файла `coverage.xml` |
| `--init [--diff\|--merge\|--force]` | жизненный цикл конфига (см. выше) |

## CLI-утилиты

`find-related-docs` и `verify-doc-links` запускают тот же сканер ссылок, из
которого строится граф — то, что карта рисует как ребро код↔доки, есть ровно
то, что они ищут и проверяют. `graph-query` отвечает на вопросы по уже
собранному снимку.

### find-related-docs

Обратный поиск: какие доки упоминают заданный файл кода. Запустите перед правкой
файла, чтобы узнать, какие страницы придётся обновить после, или подключите
`--git-added` в pre-commit-хук, чтобы недокументированные изменения помечались
до того, как попадут в репозиторий.

<details>
<summary>Флаги и примеры</summary>

```bash
find-related-docs src/mypkg/core/access.py   # один файл (голое имя тоже работает)
find-related-docs --git-added -v             # pre-commit: staged-файлы, с номерами строк в доках
find-related-docs --git-modified             # рабочее дерево: staged + unstaged изменения
```

| Флаг | Действие |
|---|---|
| `path` | файл или каталог для поиска (голое имя ищется по всему проекту) |
| `--docs-dir PATH` | каталог документации (по умолчанию: `docs`) |
| `--exclude DIRNAME` | пропустить папку с этим именем где угодно под каталогом доков (повторяемый) |
| `--git-added` | проверить все staged-файлы; также предупреждает об удалённых файлах, всё ещё упоминаемых в доках |
| `--git-modified` | проверить все изменённые файлы (staged + unstaged) |
| `-v` / `--verbose` | печатать `docs/<file>.md:<line>` для каждого упоминания |

</details>

### verify-doc-links

Проверяет, что каждая файловая ссылка в ваших `.md` указывает на реальный файл.
Коды выхода делают его готовым CI-гейтом:

<details>
<summary>Флаги и примеры</summary>

| Выход | Значение |
|---|---|
| `0` | все ссылки валидны |
| `1` | найдены битые ссылки |
| `2` | целевой путь невалиден (не найден или не `.md`-файл) |

```bash
verify-doc-links                     # весь docs/ относительно корня проекта
verify-doc-links docs/reference -v   # одно поддерево, с проблемными строками
```

```yaml
# Шаг CI (GitHub Actions)
- run: pip install graph-build
- run: verify-doc-links --root .
```

| Флаг | Действие |
|---|---|
| `path` | `.md`-файл или каталог для проверки (по умолчанию: `docs`) |
| `--root PATH` | корень проекта, относительно которого резолвятся ссылки (по умолчанию: cwd) |
| `--known-brokens PATH` | файл белого списка (по умолчанию: `<root>/known-brokens.txt`) |
| `-v` / `--verbose` | показать проблемные строки |

Помимо `known-brokens.txt`, ложные срабатывания можно глушить инлайн
HTML-комментариями (невидимыми в отрендеренном Markdown): `<!-- broken-link-ok -->`
на той же строке, `<!-- broken-links-ok-start -->` / `<!-- broken-links-ok-end -->`
вокруг блока или `<!-- ignore-ref: path/to/file.py -->` где угодно в файле.

</details>

### graph-query

Задавайте графу вопросы, не открывая браузер. Работает на JSON, записанном
через `--json` или `--compact` (определяется автоматически; по умолчанию
`docs/graph-compact.json`):

<details>
<summary>Флаги и примеры</summary>

```bash
graph-query blast-radius app/core.py   # транзитивные импортёры + каждая упоминающая их дока
graph-query hubs --top 15              # самые связанные файлы, разбивка in/out
graph-query stale-docs --check         # доки старше кода, который описывают (CI-гейт: exit 1)
graph-query orphans --type code        # файлы вообще без рёбер
```

| Команда | Отвечает на вопрос |
|---|---|
| `blast-radius <path>` | «что сломается, если тронуть этот файл» — транзитивные входящие импорты (`--depth`, `--edges` для настройки), плюс затронутые доки |
| `hubs` | «где центр тяжести» — топ узлов по сумме входящих+исходящих рёбер (`--top N`) |
| `stale-docs` | «какие доки отстают от кода» — сравнивает время последнего коммита (один проход `git log`; фолбэк на mtime), `--check` для CI |
| `orphans` | «что ни с чем не связано» — узлы со степенью 0, фильтруются по категории |

Каждая команда принимает `--json` для машиночитаемого вывода — направьте его в
`jq` или скормите агенту.

</details>

## Известные ограничения

У статического анализа есть естественные границы — граф это референсная карта,
а не семантическая:

- Динамические импорты резолвятся только для литеральных / заданных
  константами верхнего уровня имён модулей (f-строки, поиск по словарю, условное
  переприсваивание пропускаются).
- `eval` / `exec` и DI-по-строке невидимы. Точки входа `[project.scripts]` /
  `[project.gui-scripts]` в `pyproject.toml` читаются, но только чтобы вывести
  эти модули из-под пометки «мёртвый код» — рёбер они не создают.
- Шаблонизация в Markdown (`{{ ref }}`, шорткоды Jekyll/Hugo) не парсится.
- Ссылки резолвятся до целого файла — якоря секций (`file.md#part`)
  отображаются на узел файла.
- Рёбра код→код пока только для Python (слои markdown/документации
  языко-независимы).
- Один репозиторий на граф; симлинки трактуются как физические пути.
